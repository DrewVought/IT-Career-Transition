## Mission

Transition from paramedic/EMS into a professional IT role through structured, practical, documentation-driven learning. The objective is not certifications — it's building real operational competence, with a portfolio of logs and documentation that can support a resume and GitHub presence.

**Timeline:** Mid-to-late fall 2027 is a personal goal and planning target, not a deadline or a hard requirement. Current EMS employment continues until an actual job offer is in hand and the decision to transition is made. Completing all modules signals readiness to begin the active job search — not a requirement to leave immediately.

**Income target:** $90,000–105,000 base salary at entry in the NYC metro area (Jersey City/Hoboken). Financial services employers in Jersey City are the primary target sector due to above-market IT/security compensation and proximity.

**Remote work:** A long-term goal, not an entry requirement. The realistic path is hybrid (2–3 days in office) at entry, transitioning to fully remote within 12–18 months either internally or via a move to a remote-first employer. Remote becomes particularly important as a planned relocation out of the NYC metro area within the next 5+ years makes geographic flexibility valuable. Cloud and security operations roles are the natural path to remote work earliest.

## Global Rule (applies to ALL modules)

A module objective is only complete if ALL are true:

- Work is performed in a real environment (the VM, or equivalent).
- Evidence is captured as a terminal/session log.
- System behavior is observable (inputs → outputs).
- Where applicable, a full **build → test → break → fix → verify** loop exists.
- At least one task per module demonstrates independent problem solving.
- Results are reproducible from the documented steps.

> Understanding alone does not count as completion. Conceptual clarity is necessary but not sufficient — it must be demonstrated in a logged session.

## Engineering Principle

For every action taken in a lab:

1. Predict what should happen.
2. Execute the change.
3. Observe what actually happened.
4. Explain why.
5. Record the result.

Never run commands without a stated expectation of what they should do.

## Evaluation Framework

### Pre-Lab Knowledge Checks

Every new module and every new objective begins with a pre-lab knowledge check. This is standard practice — not optional. Its purpose is to establish the mental model before hands-on work begins, not to gate progress.

### Practical Demonstration Principle

When a concept is demonstrated through successful completion of a logged objective, the log is the primary evidence of understanding. Knowledge checks should not interrupt a lab to re-verify something the log will already show.

**Knowledge checks are used:**
- *Before* a lab — to build the mental model before starting.
- *During* a lab — rarely; only to prevent a misconception from taking root.
- *After* a lab — to clarify or fill gaps the log doesn't make clear.

### Repaint the Picture

If a response or explanation seems to fall short, the first move is to determine whether the issue is an **incorrect mental model** or an **unclear explanation**. If unclear, ask the learner to restate or "repaint the picture" before scoring. The score may change after clarification.

### Scoring

| Band | Meaning |
|---|---|
| 75–100% | Complete (minor terminology corrections still possible) |
| 60–74% | Needs Reinforcement — only the missing concepts are revisited |
| Below 60% | Incomplete — concept is rebuilt before progressing |

### Terminology

Corrected only when wording reflects an incorrect technical mental model. Equivalent wording is accepted without correction.

## Module 2 — System Troubleshooting

**Goal:** Diagnose and resolve system behavior issues.

1. **Process Management** — `ps`, `top`, identify process ownership, terminate/restart processes.
2. **Services** — `systemctl start/stop/restart`, interpret status; simulate or cause a service failure and recover it.
3. **Networking Troubleshooting** — diagnose connectivity failure, distinguish DNS vs. routing vs. service failure; failure → fix → verify loop.
4. **Log Analysis** — use `journalctl`/system logs, extract error signals, connect logs to system behavior.

**Complete when:** all four objectives logged, at least one multi-layer failure correctly diagnosed, logs used as the primary diagnostic tool, one fully independent troubleshooting scenario completed.

---

## Module 3 — System Administration

**Goal:** Manage multi-component system behavior.

1. Service configuration — persistent service behavior, verified across restart.
2. Permissions/policy control — manage sudo/access rules, enforce a controlled access model.
3. Storage/filesystems — mount/manage filesystem resources, persistence vs. ephemeral state.
4. Multi-layer debugging — resolve an issue spanning permissions + services + system state.

**Complete when:** system runs correctly across restarts, at least one misconfiguration created and fixed, one multi-layer dependency issue resolved.

---

## Module 4 — Security Fundamentals

**Goal:** Understand and control system exposure and access.

1. Access control review — analyze permissions system-wide, identify privilege risks.
2. Attack surface awareness — identify open ports/services, assess exposure.
3. Authentication vs. authorization — distinguish identity, permission, and access-control layers.
4. Controlled security failure — simulate a misconfiguration/exposure, detect and fix it.

**Complete when:** at least one vulnerability identified and resolved, security posture demonstrably improved.

---

## Module 5 — Networked Systems

**Goal:** Operate across multiple systems. Hardware upgrade point — dedicated lab host with Proxmox recommended before or during this module.

1. Lab host migration — move from VirtualBox on M16 to Proxmox on dedicated hardware.
2. Network segmentation — deploy pfSense or OPNsense to isolate lab traffic from home network.
3. Multi-system communication — verify connectivity across VMs.
4. Network failure analysis — diagnose DNS/routing/firewall failures across systems.
5. Distributed service awareness — understand and demonstrate service interaction across machines.

**Complete when:** multi-system issue diagnosed and resolved, network segmentation functional, cross-system communication verified.

## Module 6 — Windows Server & Active Directory

**Goal:** Administer a Windows domain environment.

1. Windows Server installation and initial configuration.
2. Active Directory domain setup — domain controller, DNS, DHCP.
3. User and group policy management — create users, groups, apply GPOs.
4. Joining a Linux machine to the Windows domain.
5. Remote administration via RDP.

**Complete when:** functional domain with at least one joined client, GPO applied and verified, remote administration demonstrated.

## Module 8 — Containers

**Goal:** Deploy and manage containerized services.

3. Deploy a real service in a container (e.g. a web server, monitoring tool, or similar).
4. Container lifecycle management — start, stop, inspect, debug a failing container.

**Complete when:** at least one real multi-container service deployed via Compose, a container failure diagnosed and resolved.

## Module 9 — Cloud Fundamentals

**AWS track:**
1. Compute — launch and administer an EC2 instance.
2. Networking — VPC, security groups, routing basics.

**Azure track:**
1. Compute — deploy and administer a VM.
2. Networking — VNet, NSGs, basics.
3. IAM — Azure AD users, RBAC.

**Complete when:** functional instances in both platforms, IAM configured correctly in both, remote administration demonstrated in both.

## Module 10 — Security Operations

**Goal:** Apply security thinking operationally across a real environment.

1. Log aggregation — centralize logs from multiple systems into one location.
2. SIEM basics — basic log querying, alert creation, event correlation.
3. Incident response simulation — respond to a simulated security event using logs as the primary source.
4. Vulnerability scanning — run a scan against a lab system, interpret results, remediate findings.
5. System hardening — apply a hardening baseline to a Linux system, document before/after posture.

**Complete when:** at least one full incident response cycle documented (detection → investigation → remediation → lessons learned), at least one hardened system with documented posture change.

## Module 11 — Capstone (Job-Ready Level)

**Goal:** Build and document one integrated environment that demonstrates operational competence across all prior modules.

The capstone is not a new learning exercise — it is the demonstration that everything built in Modules 1–10 can be combined into a coherent, real-world system.

**Required components:**
- Multiple Linux VMs with distinct roles (e.g. web server, file server, monitoring)
- Windows domain with Active Directory, DNS, DHCP
- At least one Linux machine joined to the Windows domain
- A containerized service deployed and running
- Cloud component (at minimum one instance in AWS or Azure integrated with the lab)
- Users and access control enforced across the whole environment
- Network segmentation between lab and home network
- Centralized logging and at least basic monitoring
- Security hardening applied and documented
- Backup strategy documented and tested
- Full documentation: runbook, network diagram, incident response procedure

**Independent failure recovery:** at least one scenario where an unfamiliar failure is diagnosed and resolved without guidance.

**Complete when:** full environment built, all required components functional, failure recovery demonstrated and logged, documentation complete enough that another technician could reproduce the environment.

## Parallel Certification Track

Certifications run **alongside** modules as parallel study — not as prerequisites or replacements for module work. The module builds the real competence; the certification formalizes and validates it for employers. Cert study happens on the secondary laptop (Yoga) during work downtime or low-activity periods and does not require the VM.

Certifications are tracked separately in PROGRESS.md under their own section.

### CompTIA Security+

**Begin:** During Module 4 (Security Fundamentals) — content overlaps significantly with exam domains.
**Target exam:** Around Module 5–6 completion.
**Why:** Near-mandatory for security-adjacent roles. DoD 8570 compliant, which opens government and contractor positions. Justifies the "Security Analyst" title without "Junior" in front of it, which materially affects offer numbers.
**Study resources:** Professor Messer (free), CompTIA CertMaster (paid), Darril Gibson study guide. Practice exams from Dion Training.

### AWS Cloud Practitioner

**Begin:** Early Module 9 (Cloud Fundamentals).
**Target exam:** Within first 2–3 weeks of Module 9 — this is a foundational exam, not deeply technical.
**Why:** Required stepping stone before SAA. Fast to obtain (2–4 weeks of parallel study). Establishes AWS as a formal credential on a resume.
**Study resources:** AWS Skill Builder (free tier), Stephane Maarek on Udemy.

### AWS Solutions Architect Associate (SAA-C03)

**Begin:** Mid-Module 9, after Cloud Practitioner is obtained.
**Target exam:** Around Module 10 completion.
**Why:** The AWS cert that actually changes offer numbers. Recognizable to hiring managers across financial services, tech, and government sectors. Pairs with the portfolio's cloud work to make the combination credible.
**Study resources:** Stephane Maarek on Udemy, Tutorials Dojo practice exams (considered the best SAA practice available), AWS documentation.

### Optional additions (if pace and timeline allow)

These are not required for the primary transition goal but are worth knowing about:

- **CompTIA Linux+** — validates the Module 1–3 work formally. Low additional study burden given the hands-on experience already built. Good if a SysAdmin-primary path is the target.
- **Azure Fundamentals (AZ-900)** — quick companion to AWS Cloud Practitioner during Module 9. Adds Azure credibility with minimal additional effort since Module 9 covers both platforms.
- **CompTIA Network+** — useful if the networking/infrastructure path becomes primary over security. Less urgent given the portfolio already covers networking operationally.
